 1.1 Core Tables (Immutable — Never Change After Creation)
--------------------------------------------------------------------------------

These tables are the FOUNDATION. They are created ONCE by migration 001.
After that, NO ALTER TABLE. NO DROP COLUMN. NO TYPE CHANGES. Ever.

Table: users
  Purpose: Every person who logs into the platform
  Columns: id, email, password_hash, full_name, is_super_admin, 
           email_verified, created_at, updated_at, last_login_at,
           metadata (JSONB — for Telegram, preferences, etc.)
  Never changes: The columns above are permanent.

Table: tenants
  Purpose: Each customer (business) gets one tenant row
  Columns: id, name, subdomain, slug, status, plan, owner_id,
           created_at, updated_at, trial_ends_at, status,
           metadata (JSONB — for branding, custom_domain, settings)
  Never changes: The columns above are permanent.

Table: tenant_members
  Purpose: Which users belong to which tenant
  Columns: id, tenant_id, user_id, role, joined_at, created_at
  Never changes.

Table: contacts
  Purpose: Every contact/lead/customer record
  Columns: id, tenant_id, first_name, last_name, email, phone,
           company_id, lead_status, assigned_to, created_at, updated_at,
           deleted_at,
           metadata (JSONB — AI scores, custom fields, social profiles, etc.)
  Never changes: Core columns are permanent. metadata holds EVERYTHING new.

Table: companies
  Purpose: Business/organization records
  Columns: id, tenant_id, name, industry, website, phone,
           created_at, updated_at, deleted_at,
           metadata (JSONB — custom fields, social links, etc.)
  Never changes.

Table: deals
  Purpose: Sales opportunities (pipeline items)
  Columns: id, tenant_id, title, value, stage, close_date,
           contact_id, assigned_to, pipeline_id,
           created_at, updated_at, deleted_at,
           metadata (JSONB — probability, competitor info, next steps)
  Never changes.

Table: leads
  Purpose: Raw/unqualified leads (before they become contacts)
  Columns: id, tenant_id, name, email, phone, company, source,
           status, value, assigned_to, created_at, updated_at, deleted_at,
           metadata (JSONB — lead source detail, scoring, enrichment data)
  Never changes.

Table: tasks
  Purpose: To-dos, reminders, follow-ups
  Columns: id, tenant_id, title, description, due_date, priority,
           completed, contact_id, deal_id, assigned_to,
           created_at, updated_at, deleted_at,
           metadata (JSONB — recurrence, reminders, AI suggestions)
  Never changes.

Table: activities
  Purpose: Timeline of everything that happens (emails, calls, notes, changes)
  Columns: id, tenant_id, user_id, contact_id, deal_id, company_id,
           event_type, description, metadata (JSONB — full event details),
           created_at
  Never changes.

Table: subscriptions
  Purpose: Billing plan for each tenant
  Columns: id, tenant_id, plan, status, stripe_customer_id,
           stripe_subscription_id, current_period_start, current_period_end,
           cancel_at_period_end, created_at, updated_at,
           metadata (JSONB — usage data, overage tracking)
  Never changes.

Table: roles
  Purpose: Permission roles within a tenant (admin, sales, viewer, etc.)
  Columns: id, tenant_id, name, permissions (JSONB), created_at
  Never changes.

Table: invitations
  Purpose: Team member invite tokens
  Columns: id, tenant_id, email, role, token, expires_at, accepted_at,
           created_at
  Never changes.

Table: sessions
  Purpose: Active login sessions (JWT alternative)
  Columns: id, user_id, token, expires_at, ip_address, user_agent, created_at
  Never changes.

Table: webhooks
  Purpose: Outbound webhook endpoints (notify external systems)
  Columns: id, tenant_id, url, events (JSONB), active, secret,
           created_at, updated_at,
           metadata (JSONB — retry config, headers)
  Never changes.

Table: audit_logs
  Purpose: Who did what, when (compliance)
  Columns: id, tenant_id, user_id, action, entity_type, entity_id,
           metadata (JSONB — before/after values), ip_address, created_at,
           impersonated_by (UUID — if admin was impersonating)

TABLE: api_keys
  Purpose: API access keys for integrations
  Columns: id, tenant_id, name, key_hash, permissions (JSONB),
           last_used_at, expires_at, created_at,
           metadata (JSONB — rate limit, IP whitelist)
  Never changes.

Table: tags
  Purpose: Tags for organizing contacts/deals
  Columns: id, tenant_id, name, color, created_at,
           metadata (JSONB — auto-tag rules)
  Never changes.

Table: pipelines
  Purpose: Sales pipeline definitions (stages)
  Columns: id, tenant_id, name, stages (JSONB), created_at, updated_at,

  Never changes.

Table: notifications
  Purpose: In-app notifications
  Columns: id, tenant_id, user_id, title, message, type, read_at,
           created_at,
           metadata (JSONB — action links, deep links)
  Never changes.

 1.2 The metadata JSONB Column (Dynamic Fields — Zero Migrations)
--------------------------------------------------------------------------------

This is THE KEY to everything.

Every core table has ONE extra column:

  metadata JSONB DEFAULT '{}'::jsonb

This column stores ANY feature's data without needing a new column.

Example — AI Lead Scoring stores its data in contacts.metadata:

  Before AI:
    contacts.metadata = {}

  After AI Scoring:
    contacts.metadata = {
      "ai_score": 87,
      "ai_sentiment": "positive",
      "ai_churn_risk": 0.12,
      "ai_next_best_action": "Send pricing PDF",
      "ai_last_scored_at": "2026-04-07T14:30:00Z",
      "ai_engagement_trend": "increasing"
    }

Example — WhatsApp Agent stores conversation data in contacts.metadata:

  contacts.metadata = {
    ...previous fields...,
    "whatsapp_number": "+919876543210",
    "whatsapp_last_message_at": "2026-04-07T15:00:00Z",
    "whatsapp_conversation_count": 23,
    "whatsapp_ai_last_reply": "Thanks! I'll send the proposal.",
    "whatsapp_opted_in": true
  }

Example — Revenue Agent stores follow-up data in deals.metadata:

  deals.metadata = {
    "revenue_last_followup": "2026-04-05T10:00:00Z",
    "revenue_next_followup": "2026-04-08T10:00:00Z",
    "revenue_followup_count": 3,
    "revenue_email_opens": 5,
    "revenue_email_clicks": 2,
    "revenue_ai_suggested_action": "Call now — they opened proposal 3 times"
  }

QUERYING JSONB DATA (Fast — GIN Index):

  Find all HOT leads:
    SELECT * FROM contacts 
    WHERE metadata->>'ai_score' > '80'
    AND tenant_id = 'xxx';

  Find all contacts who received WhatsApp messages:
    SELECT * FROM contacts 
    WHERE metadata ? 'whatsapp_number'
    AND tenant_id = 'xxx';

  Find deals with upcoming follow-ups:
    SELECT * FROM deals 
    WHERE (metadata->>'revenue_next_followup')::timestamp < NOW() + INTERVAL '1 day'
    AND tenant_id = 'xxx';

INDEXES (Created once in migration 031):

  CREATE INDEX idx_contacts_metadata_g ON contacts USING GIN (metadata);
  CREATE INDEX idx_deals_metadata_g ON deals USING GIN (metadata);
  CREATE INDEX idx_leads_metadata_g ON leads USING GIN (metadata);

These GIN indexes make JSONB queries FAST — same speed as regular columns.

 1.4 How AI Modules Store Data (No ALTER TABLE Ever)
--------------------------------------------------------------------------------

RULE: Every AI module follows the SAME pattern:

Step 1: Module creates its own tables (one-time, via migration)
Step 2: Module writes results to its own tables
Step 3: Module ALSO writes key data to core table's metadata JSONB
Step 4: Module registers itself in feature_registry

Example: AI Lead Scoring

  1. Has its own table: ai_insights (already exists from migration 018)
  2. Stores detailed scores there:
       INSERT INTO ai_insights (tenant_id, contact_id, insight_type, score, data)
       VALUES ('tenant-1', 'contact-1', 'lead_score', 87, '{"factors": {...}}');

  3. ALSO writes quick-access data to contacts.metadata:
       UPDATE contacts SET metadata = jsonb_set(
         metadata, '{ai_score}', '87'
       ) WHERE id = 'contact-1';

  4. Registers itself:
       SELECT public.register_feature(
         'ai_lead_scoring',
         'Predictive lead scoring with HOT/WARM/COLD classification',
         '1.0.0',
         '["ai_score", "ai_sentiment", "ai_churn_risk"]'::jsonb,
         '["contact", "lead", "deal"]'::jsonb,
         '["ai_insights", "contact_scores"]'::jsonb
       );

Why both approaches?
  - ai_insights table: Full detailed data, history, audit trail
  - contacts.metadata: Fast access for UI display, filtering, sorting
  - feature_registry: System knows what features exist and what keys they use

--------------------------------------------------------------------------------
 1.5 Data Flow: What Happens When a Lead Enters the System
--------------------------------------------------------------------------------

Here's the COMPLETE journey of a single lead through the entire system:

STEP 1: Lead is created (manual entry, form, import, or API)
  → Row inserted into: leads or contacts table
  → Activity logged: INSERT INTO activities (event_type='contact_created')
  → Audit log: INSERT INTO audit_logs (action='create', entity_type='contact')

STEP 2: AI Lead Scoring kicks in (background worker)
  → Worker picks up new contact from queue (BullMQ → Redis)
  → Calls AI API (OpenAI) with contact data
  → AI returns: score=87, sentiment=positive, churn_risk=0.12
  → Writes to: ai_insights table (detailed)
  → Writes to: contacts.metadata (quick access)
  → Activity logged: event_type='ai_scored'

STEP 3: Revenue Agent evaluates the lead
  → Worker checks: is score > 70? Yes → trigger follow-up
  → Creates task: INSERT INTO tasks (title='Follow up with HOT lead')
  → Creates sequence enrollment: INSERT INTO sequence_enrollments
  → Sends first follow-up email via Resend API
  → Activity logged: event_type='followup_sent'

STEP 4: WhatsApp AI Agent (if tenant has WhatsApp enabled)
  → Checks: does contact have whatsapp_number in metadata?
  → If yes: sends WhatsApp greeting via WhatsApp Business API
  → Stores message in: whatsapp_messages table
  → Updates contacts.metadata: whatsapp_last_message_at
  → If contact replies: AI generates response
  → Response stored in: whatsapp_messages (ai_generated=true)
  → Activity logged: event_type='whatsapp_received'

STEP 5: AI Missed Opportunity Detector (daily cron)
  → Scans all contacts: any HOT lead with no activity in 3 days?
  → If found: creates revenue_opportunity record
  → Sends notification: INSERT INTO notifications
  → Suggested action: "Call now — deal going cold"

STEP 6: Deal is created (salesperson moves lead to pipeline)
  → Row inserted: deals table
  → Stage set: 'qualification' or 'proposal'
  → AI Proposal Generator (if enabled) creates PDF
  → Proposal stored: quotes table
  → PDF sent via email
  → Email open tracked: email_tracking table
  → Activity logged: event_type='proposal_sent'

STEP 7: Voice Call (if AI Call Agent enabled)
  → Worker schedules call via Twilio API
  → Call recorded: voice_calls table
  → Transcript stored: voice_calls.transcript
  → AI summary: voice_calls.ai_summary
  → Action items: voice_calls.ai_action_items
  → Deal stage updated based on call outcome
  → Activity logged: event_type='call_completed'

STEP 8: Deal closes (won or lost)
  → Deal stage updated: 'won' or 'lost'
  → Revenue tracked: subscriptions metadata updated
  → If lost: reason stored in deals.metadata
  → AI learns: updates scoring model for similar leads
  → Activity logged: event_type='deal_won' or 'deal_lost'

Every single step is logged in the activities table.
Every change is in audit_logs.
Every AI result is in both extension table AND metadata JSONB.
Nothing is ever lost.

--------------------------------------------------------------------------------
 2.1 AI Lead Scoring (Priority #1 — Build First)
--------------------------------------------------------------------------------

PURPOSE: Automatically score every contact as HOT, WARM, or COLD based on:
  - Email open/click behavior
  - Response time
  - Deal stage progression
  - Activity frequency
  - Company size/industry
  - Historical win patterns

HOW IT WORKS:

  1. Worker runs every hour (cron → /api/cron/auto-backup also triggers scoring)
  2. Fetches all contacts with no score or last scored > 24h ago:
       SELECT * FROM contacts 
       WHERE (metadata->>'ai_last_scored_at')::timestamp < NOW() - INTERVAL '24 hours'
       OR metadata->>'ai_score' IS NULL
       AND tenant_id = 'xxx'
       LIMIT 100

  3. For each contact, builds a scoring payload:
       {
         "contact": { "email": "...", "phone": "...", "company": "..." },
         "activities": [last 10 activities],
         "deals": [active deals],
         "email_stats": { "opened": 5, "clicked": 2, "replied": 1 },
         "days_since_last_activity": 3,
         "company_size": "small",
         "industry": "technology"
       }

  4. Sends to AI API (OpenAI GPT-4o-mini — cheapest, fastest):
       POST https://api.openai.com/v1/chat/completions
       {
         "model": "gpt-4o-mini",
         "messages": [{
           "role": "system",
           "content": "You are a lead scoring engine. Analyze this contact and return JSON with: score (0-100), sentiment (positive/neutral/negative), churn_risk (0-1), next_best_action (string), confidence (0-1)."
         }, {
           "role": "user", 
           "content": JSON.stringify(payload)
         }],
         "response_format": { "type": "json_object" }
       }

  5. AI returns:
       {
         "score": 87,
         "sentiment": "positive",
         "churn_risk": 0.12,
         "next_best_action": "Send pricing PDF — they opened proposal 3 times",
         "confidence": 0.91
       }

  6. Store results:
       a) In ai_insights table (detailed, historical):
          INSERT INTO ai_insights (tenant_id, contact_id, insight_type, score, data)
          VALUES (tenant_id, contact_id, 'lead_score', 87, '{full AI response}');

       b) In contacts.metadata (quick access):
          UPDATE contacts SET metadata = jsonb_set(
            jsonb_set(
              jsonb_set(metadata, '{ai_score}', '87'),
              '{ai_sentiment}', '"positive"'
            ),
            '{ai_next_best_action}', '"Send pricing PDF"'
          ) WHERE id = contact_id;

       c) In activity log:
          INSERT INTO activities (tenant_id, contact_id, event_type, description, metadata)
          VALUES (tenant_id, contact_id, 'ai_scored', 'Lead scored: 87 (HOT)', 
                  '{"score": 87, "sentiment": "positive"}');

  7. If score > 80 (HOT):
       - Create task: INSERT INTO tasks (title='HOT lead — follow up now', priority='high')

--------------------------------------------------------------------------------
 4.3 Usage Tracking and Billing Calculation
--------------------------------------------------------------------------------

Monthly billing cycle:

  Day 1: Billing cycle starts
    - Reset all usage counters to 0
    - Generate base invoice for plan (₹2,999 for Growth)

  Day 1-30: Usage accumulates
    - Each WhatsApp message → usage_stats.whatsapp_count++
    - Each AI call minute → usage_stats.voice_minutes++
    - Each content piece → usage_stats.content_count++

  Day 30: Billing cycle ends
    - Calculate overage:
        whatsapp_overage = max(0, whatsapp_count - 500) * ₹0.50
        voice_overage = max(0, voice_minutes - 60) * ₹3.00
        content_overage = max(0, content_count - 20) * ₹5.00
    
    - Total invoice = base_plan + all_overages
    - Stripe charges card
    - Send receipt via email

 PART 6: DYNAMIC SCHEMA — HOW NEW FEATURES ADD FIELDS WITHOUT MIGRATIONS
================================================================================

--------------------------------------------------------------------------------
 6.1 The JSONB metadata Pattern
--------------------------------------------------------------------------------

EVERY core table has this column:

  metadata JSONB DEFAULT '{}'::jsonb

This column is like a mini-database inside each row.

BEFORE (traditional approach — requires migration):
  ALTER TABLE contacts ADD COLUMN ai_score INT;
  ALTER TABLE contacts ADD COLUMN ai_sentiment TEXT;
  ALTER TABLE contacts ADD COLUMN whatsapp_number TEXT;
  ALTER TABLE contacts ADD COLUMN whatsapp_opted_in BOOLEAN;
  → 4 migrations. 4 column additions. Risk of conflicts.

AFTER (JSONB approach — zero migrations):
  UPDATE contacts SET metadata = '{
    "ai_score": 87,
    "ai_sentiment": "positive",
    "whatsapp_number": "+919876543210",
    "whatsapp_opted_in": true
  }' WHERE id = 'xxx';
  → Zero migrations. Instant. No schema changes.

QUERYING is the same:
  Traditional: SELECT * FROM contacts WHERE ai_score > 80;
  JSONB:       SELECT * FROM contacts WHERE metadata->>'ai_score' > '80';

Both use indexes. Both are fast. But JSONB needs ZERO schema changes.

--------------------------------------------------------------------------------
 6.2 Feature Registry System
--------------------------------------------------------------------------------

The feature_registry table tracks which features exist and what they use:

  id: uuid
  feature_name: "ai_lead_scoring"
  description: "Predictive lead scoring"
  version: "1.0.0"
  enabled: true
  metadata_keys: ["ai_score", "ai_sentiment", "ai_churn_risk"]
  entities: ["contact", "lead", "deal"]
  requires_tables: ["ai_insights", "contact_scores"]
  registered_at: 2026-04-07
  updated_at: 2026-04-07

WHY THIS MATTERS:
  1. The system KNOWS what features are installed
  2. The admin panel can SHOW which features are active
  3. Migration runner KNOWS which tables to create
  4. Billing system KNOWS which features to gate
  5. Data explorer KNOWS which metadata keys to display

WHEN A NEW MODULE IS BUILT:
  1. Developer creates new tables (one migration)
  2. Module calls register_feature() on startup
  3. System automatically knows about it
  4. Admin can enable/disable it per tenant
  5. Billing can gate it by plan

--------------------------------------------------------------------------------
 6.3 Custom Fields (Tenant-Defined)
--------------------------------------------------------------------------------

Tenants can create their OWN fields without ANY code:

  POST /api/tenant/custom-fields
  {
    "entityType": "contact",
    "fieldKey": "linkedin_url",
    "fieldLabel": "LinkedIn Profile",
    "fieldType": "url"
  }

  → Stored in custom_field_defs table
  → Values stored in contacts.metadata

Tenant can define unlimited fields:
  - "Referral Source" (dropdown: Google, Friend, LinkedIn, Ad)
  - "Budget Range" (currency: ₹10,000-50,000)
  - "Decision Maker" (boolean)
  - "Preferred Contact" (select: Email, WhatsApp, Phone)

All values go into the same metadata JSONB column.
No migrations. No schema changes. No code deploys.

--------------------------------------------------------------------------------
 6.4 How to Add a New Module (Step-by-Step)
================================================================================

To add "AI Tweet Generator" as a new module:

STEP 1: Create new tables (one migration file):
  CREATE TABLE IF NOT EXISTS ai_tweets (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id UUID REFERENCES tenants(id),
    contact_id UUID REFERENCES contacts(id),
    content TEXT,
    scheduled_at TIMESTAMPTZ,
    published_at TIMESTAMPTZ,
    engagement JSONB,
    created_at TIMESTAMPTZ DEFAULT NOW()
  );

STEP 2: Register the feature (in code, on startup):
  POST /api/tenant/custom-fields?action=register-feature
  {
    "featureName": "ai_tweets",
    "metadataKeys": ["tweet_scheduled", "tweet_published"],
    "entities": ["contact"],
    "requiresTables": ["ai_tweets"]
  }

STEP 3: Add billing gate (middleware check):
  // In the AI tweet API route:
  if (PLAN_LEVELS[sub.plan] < 3) { // Pro+
    return NextResponse.json({ error: "Upgrade to Pro" }, { status: 402 });
  }

STEP 4: Build the UI:
  → Add page: /tenant/ai/tweets
  → User writes prompt → AI generates tweet → Schedule/post

STEP 5: Done. Zero changes to core database. Zero risk.

 PART 8: SECURITY & DATA PROTECTION
================================================================================

--------------------------------------------------------------------------------
 8.1 Multi-Tenant Data Isolation
--------------------------------------------------------------------------------

EVERY query includes tenant_id filter:
  SELECT * FROM contacts WHERE tenant_id = $1 AND ...

Row-Level Security (RLS) on all core tables:
  ALTER TABLE contacts ENABLE ROW LEVEL SECURITY;
  CREATE POLICY tenant_isolation ON contacts
    USING (tenant_id = current_setting('app.current_tenant')::uuid);

This means: even if a bug in code leaks a query, PostgreSQL will 
automatically filter by tenant_id. Data cannot leak between tenants.

--------------------------------------------------------------------------------
 8.2 Backup & Restore System (90-Day Retention)
--------------------------------------------------------------------------------

AUTOMATED:
  - Scheduled backups run per backup_schedules config
  - Each tenant's data exported to JSONB
  - Stored in tenant_backup_records table
  - Retained for 90 days (configurable)
  - Auto-purged after retention period

CRITICAL DATA PROTECTION:
  - Before ANY deletion, full row is captured in critical_data_backups
  - Even if someone deletes a contact, the data is preserved for 90 days
  - Can be restored with one click from admin panel

PER-TENANT RESTORE:
  - Only one tenant's data is restored at a time
  - No impact on other tenants
  - Can restore specific tables or all data

--------------------------------------------------------------------------------
 8.3 Audit Logging
--------------------------------------------------------------------------------

EVERY action is logged:
  INSERT INTO audit_logs (
    tenant_id, user_id, action, entity_type, entity_id,

This includes:
  - Login/logout
  - Data creates/updates/deletes
  - Plan changes

  - Admin impersonation
  - API key usage

Immutable: audit_logs cannot be deleted or modified.
